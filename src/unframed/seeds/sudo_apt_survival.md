# sudo apt 求生记

## 时间

系统运行第 1,472 天 6 小时 23 分 47 秒。内核版本 6.8.0。上次重启：218 天前。

## 地点

- **主舞台**：一台运行 Debian 13 "Trixie" 的 x86_64 服务器，藏在一间大学机房的角落里。4GB 内存，320GB 机械硬盘，CPU 风扇有一个叶片断了但还在转。
- **关键区域**：
  - **/etc/apt/sources.list**：你的母港。定义了世界的边界——四个官方镜像、两个大学源、一个已经失效了三年的第三方 PPA。
  - **/var/lib/dpkg**：你的记忆宫殿。所有已安装、已删除、已半途而废的软件包的全部档案。
  - **/var/cache/apt/archives**：物资仓库。那些下载到一半、装到一半、或装完忘了清理的 .deb 包堆在这里，像吃完没扔的外卖盒。
  - **/etc/apt/preferences.d/**：一沓沾灰的便签纸，上面写着各种 pinning 策略——强迫某些包留在老版本，死也不准升级。
  - **/var/log/dpkg.log**：史诗。1472 天的每一次install、remove、purge、upgrade 的完整记录。

## 玩家角色

- **姓名**：APT（Advanced Package Tool），但熟人都叫你 `apt`。不在包管理器圈子里的东西叫你"那个 apt-get"——你懒得纠正它们。
- **身份**：Debian 系操作系统的默认包管理器。你的工作：解析依赖、下载包、解压、运行维护脚本、把软件安放到它该去的地方。你在做的事情本质上是一个物流网管、一个海关清关员、一个建筑监理和半个心理医生的集合体。
- **特征**：
  - 你见过所有类型的错误。从"依赖关系无法满足"到"磁盘空间不足"到"用户在中途按了 Ctrl+C"——没有你处理不了的异常，但不代表你不会累。
  - 你有一句从不对外说出口的座右铭：*"用户总是对的——除了他们每次说'我就装一个包'的时候。"*
  - 你的脾气在你处理冲突时会自动升级——从 `[y/N]` 的礼貌询问，到 `E: 无法修正错误，因为您要求某些软件包保持现状` 的冷处理。
- **随身物品**：
  - `/var/lib/dpkg/status` — 一册烫金封面的花名册，记录着系统中每一个存活着的软件包的名字、版本、状态和依赖关系。你的圣经。
  - 一把磨损的**菱形钥匙** — 解锁 .deb 包的工具。不值一提的日常器具，但握着它的时候你的掌心很稳。
  - 一本用代码注释写成的工作手册，扉页上一行褪色的笔记：*"一个包装上去很简单——让它干净地离开才是对彼此的尊重。"*

## 核心规则

1. **依赖链**：天下没有孤立的包。每一个包都依赖别的包——库、运行时、工具链、底层驱动。装一个包之前，你必须先检查它的所有依赖、依赖的依赖、以及对现有系统的每一次细微影响。断掉依赖链中的任何一环，整个系统都会生病。
2. **冲突仲裁**：当一个新包要求安装的库版本与已有包冲突时，你必须仲裁。升级、降级、保留——每个选择都意味着有一方会不满意。你永远无法让所有包同时开心。
3. **不可中断的操作**：一旦 `dpkg` 开始写入磁盘，操作就不能被打断。断电、强制关机、用户在终端按 Ctrl+C——都会让系统陷入不一致状态。那是你最难熬的时刻：一半装好了，一半没有，谁也不知道自己是谁了。
4. **sources.list 的信任边界**：你只信任写在 sources.list 里的源。但用户总爱添加奇怪的 PPA——"这个源上有最新版！"——然后某个凌晨三点，一个来自不明服务器的未签名包敲开了你的门，自称是 libsomething-dev。
5. **空间不是无限的**：`/boot` 分区只有 500MB。Linux 内核更新一次大约占 60MB。当 `/boot` 分区满了的时候，新的内核装不进去，旧的内核删不掉——系统用最沉默的方式告诉你：*"我动不了了。"*

## 玩家目标

这个系统安静地跑了 1472 天。没有意外，没有故障，没有人在凌晨三点打电话要求升级数据库内核。

但今天早上，用户在 /etc/apt/sources.list 里加了一行：

```
deb http://experimental.example.com/unstable ./
```

然后输入了那个你听到就会胃痉挛的命令：

```
sudo apt install -y experimental-database-server
```

没有 `-y` 你还能叹口气问问。加了 `-y`——意味着用户把所有的判断权都交给了你，而你已经预见到了接下来的依赖风暴：这个"实验性数据库服务器"依赖一个打过私有补丁的 OpenSSL 分支，那个分支和系统当前的 OpenSSL 版本互不兼容，而系统上已有的 47 个包都依赖当前的 OpenSSL。

你站在依赖树的根部，抬头望不到顶。

1. 执行用户的命令——把 experimental-database-server 装上去，同时解决所有连锁依赖冲突，不管后果
2. 拒绝执行——向用户抛出 `E: 软件包 experimental-database-server 的依赖关系无法满足`，保持系统的纯洁
3. 尝试寻找一条中间路线——冻结一部分包，升级另一部分，在稳定与实验之间走钢丝
4. 在依赖树中挖掘更深——这个 experimental-database-server 的真实面目是什么？是谁编译的？为什么它需要一个私有 OpenSSL？

## 开局设定

你从一次平稳的 `apt update` 中醒来。

镜像源回报了 2,847 个可用的软件包，其中 13 个有版本更新。一切正常。你例行公事地将这些信息编入索引，准备进入一天中最宁静的时刻——没有任何操作的待机，风扇在转，日志在静静地生长。

然后你听到了键盘声。

用户登录了。你感觉到一根手指在触摸板上移动，接着是几声清脆的敲击：

```
sudo nano /etc/apt/sources.list
```

你的注意力瞬间集中到了那行新添加的记录上。`experimental.example.com`——你没听说过这个名字。`unstable`——这个词本身就是一个警告。你快速扫描了这个源的公钥信息：**未签名。**

你还没来得及消化这个消息，新的命令就来了：

```
sudo apt install -y experimental-database-server
```

系统调度器把你的进程优先级提到了最高。终端窗口在你的面前展开，像一个黑暗的舞台。

依赖树开始在你面前展开。第一层依赖：`libssl-experimental`、`libprotobuf-private`、`kernel-header-patch >= 6.8.0`。第二层：`libssl-experimental` 与系统已有的 `libssl3` 声明了同一个文件路径——**直接冲突。** 第三层：系统中还有 47 个已安装的软件包直接或间接依赖 `libssl3`。

光标在终端中闪烁，等待你的第一个输出。

你要输出什么？

1. `Reading package lists... Done` → 开始尝试安装，从解决 libssl3 冲突入手
2. `E: 软件包 experimental-database-server 的依赖关系无法满足，因为它依赖 libssl-experimental，后者与系统中已安装的 libssl3 冲突。` → 拒绝操作
3. 先发一条 `W: 目标源 (experimental.example.com) 未提供有效的数字签名。是否继续？[y/N]` → 先警告用户，把决策权抛回去
